# Python AI Template — 使用手册

> 面向**人类开发者**与**任意 AI Agent 工具**（Cursor、Claude Code、Copilot Chat、自建 Bot 等）协作时的操作说明。  
> 契约与架构细节以 `docs/` 为准；本手册侧重「怎么跑起来、怎么按工单推进、何时开新会话」。

---

## 1. 这是什么

| 项 | 说明 |
|----|------|
| **定位** | 脱离具体业务的 AI 平台模板（Chat、RAG、Agent、Workflow、SSE 等） |
| **开发模式** | Harness 驱动：docs → 工单 F01–F21 → pytest 验证 → 更新 `feature_list.json` |

| **老项目参考** | 子目录 `TestNewHarness/`（可选，见 `docs/00-meta/DEPENDENCY_MAP.md` 表 3） |

## 2. 环境要求

- Python **3.11+**
- Docker Desktop（PostgreSQL、Redis、Qdrant；F02 起需要 PG/Redis）
- Git
- Windows 推荐 **PowerShell**；也可用 Git Bash / WSL 跑 `init.sh`

## 3. 第一次安装

```powershell
cd C:\path\to\python-ai-template

# 1. 依赖
pip install -e ".[dev]"

# 2. 配置（根目录，与 pyproject.toml 同级）
#    本仓 .env 纳入 Git；可按需从 TestNewHarness\.env 合并键名
notepad .env

# 3. 基础设施
docker compose up -d postgres redis
# 需要 Qdrant 时：docker compose up -d qdrant

# 4. 健康检查
.\init.ps1

# 5. 启动 API（开发）
uvicorn app.main:app --host 0.0.0.0 --port 6006 --reload
```

浏览器访问：<http://localhost:6006/api/v1/health>，应返回 `code: 0`。

## 4. 配置与密钥注入

与 **TestNewHarness** 相同的三层方式（实现见 `app/core/config.py`）。

### 4.1 优先级（高 → 低）

1. `secrets/` 目录文件 → 容器内 `/run/secrets/<文件名>`
2. 仓库根目录 **`.env`**
3. `configs/override.yaml`（本地覆盖，默认不进 Git）
4. `configs/default.yaml`

### 4.2 `.env` 放哪

**路径**：`python-ai-template/.env`（不是 `app/` 下）。

常用变量见 `.env.example`，例如：

| 变量 | 用途 |
|------|------|
| `DATABASE_URL` | PostgreSQL |
| `REDIS_URL` | Redis |
| `QDRANT_URL` | 向量库 |
| `TEXT_MODEL__QWEN_API_KEY` | LLM（F04 后用） |
| `SECURITY_API_KEY` | API Key 认证（F20） |

修改后需**重启** uvicorn 或 `docker compose restart app`。

### 4.3 Docker Secrets（可选）

```powershell
mkdir secrets
Set-Content secrets\qwen_api_key -Value 'sk-xxx' -NoNewline
Set-Content secrets\api_key -Value 'your-api-key' -NoNewline
docker compose up -d
```

说明见 `secrets/README.md`。`compose` 已挂载 `./secrets:/run/secrets:ro`。

### 4.4 从老项目迁移配置

```powershell
# 可选：复制后检查键名（新模板已兼容 TEXT_MODEL__* 等）
copy TestNewHarness\.env .env
```

对照 `DEPENDENCY_MAP.md` **契约漂移表**，路径/API 以本仓 `docs/01-architecture/API_CONTRACT.md` 为准。

## 6. 从零生成项目：一步一步（你现在的状态）

> **当前进度**：F01 已通过；下一张工单是 **F02**（见 `progress.md`）。下面按「人类操作 + Agent 操作」分开写。

### 6.0 开工前（人类做一次）

| 步 | 你做什么 | 通过标准 |
|----|----------|----------|
| A1 | `cd` 到仓库根目录 | 能看到 `使用手册.md`、`app/` |
| A2 | `pip install -e ".[dev]"` | 无报错 |
| A3 | 编辑根目录 `.env`（数据库/Redis 等） | 与 compose 一致 |
| A4 | `docker compose up -d postgres redis` | 容器 healthy |
| A5 | `.\init.ps1` | 7/7 通过 |
| A6 | 打开 `progress.md` 确认 `active_feature_id` = **F02** | 与 handoff 一致 |

### 6.1 用任意 Agent 工具：第一张工单怎么跑（F02 示例）

**原则**：工具无关；每次只推进 **一条工单**；话术以仓库文件为准，不要口头改规则。

#### 会话 ① — 阶段 1（只读，不写代码）

1. **开新会话**（见 §8 何时开新会话）。
2. 把下面整段粘贴给 Agent（或直接复制 `session-handoff.md`「下一会话开场白」）：

```text
按 AGENTS.md 0.1 启动，执行 progress.md 的 active 任务（工单 F02）。
先运行 init.ps1，再读 FACT_REGISTRY、ARCHITECTURE、DATA_MODEL、PERSISTENCE_LAYER。
可选对照 DEPENDENCY_MAP.md 表 3（F02 无老路径则跳过）；禁止整文件复制。
严格阶段 1 只读：首段约束摘要 ≥5 条，输出工单 §1.3 交付物（差异表+文件清单），禁止写代码。
```

3. **你审阅** Agent 输出的阶段 1 交付物：差异是否合理、范围是否只在 F02。
4. 满意则回复（开**同一会话**即可）：

```text
阶段1通过，进入阶段2，严格按工单 F02 的 §2.1–2.3 与 §2.2 实现清单创建文件。
```

#### 会话 ① 续 — 阶段 2（写代码）

5. Agent 实现 `工单/F02_*.md` §2.2 列出的文件 + `tests/test_02_database.py`。
6. 你在本机执行工单验证命令：

```powershell
pytest tests/test_02_database.py -v
alembic upgrade head
.\init.ps1
bash scripts/check-architecture.sh   # 有 Git Bash/WSL 时
```

7. 全部通过 → 让 Agent 做 **阶段 3**（或你换工具也行，见 §8.3）：

```text
阶段2验证已通过。执行阶段3：对照 evaluator-rubric 自审，更新 feature_list.json（F02 passing+evidence）、progress.md（下一任务 F03）、session-handoff.md（更新下一会话开场白为 F03 阶段1）。
```

#### 会话 ② — 下一张工单 F03

8. **建议开新会话**（F02 已 closing）。
9. 只复制 **更新后的** `session-handoff.md`「下一会话开场白」（应已指向 F03）。
10. 重复：阶段 1 → 你确认 → 阶段 2 → 你跑 pytest → 阶段 3。

### 6.2 后续工单（F04 → F21）重复节奏

每张工单同一节奏，**不要跳号**（除非 `DEPENDENCY_MAP` 允许且前序已 passing）：

```
开新会话（推荐）→ 粘贴 handoff 开场白 → 阶段1 → 你:阶段1通过 → 阶段2 → 你:跑验证 → 阶段3 → 更新 handoff
```

工单顺序见 §7。单工单上下文过长（实现文件很多）时，阶段 2 与阶段 3 可拆成两个会话（§8.2）。

### 6.3 你该盯的文件（所有工具通用）

| 文件 | 作用 |
|------|------|
| `progress.md` | 当前在做哪张工单 |
| `feature_list.json` | F01–F21 是否 passing |
| `session-handoff.md` | **下一会话复制粘贴的开工话术** |
| `工单/Fxx_*.md` | 本迭代实现范围与 §2.2 文件清单 |
| `docs/00-meta/DEPENDENCY_MAP.md` | 依赖与 TestNewHarness 对照（表 3） |
| `AGENTS.md` | 给 Agent 的强制协议（可附：「遵守 AGENTS.md」） |

换工具时：**不要**依赖上一款工具的聊天记录；只依赖上述文件 + 本机 git 状态。

## 7. 后续改功能 / 新增接口（三条路径）

先判断属于哪一类：

| 你的需求 | 走哪条路 | 能否口头让 Agent 直接改 |
|----------|----------|------------------------|
| 按计划实现 **尚未 passing** 的能力 | **工单 FNN**（§6） | 可以，但必须阶段 1→2→3 |
| **修改已 passing** 的功能/逻辑 | **变更单 CO-NNN** | 不可以，必须变更流程 |
| **新增** 计划外的大能力 | 先补 docs → 新工单或 F22+ | 不可以跳过 docs |

### 7.1 修改已有功能（已 passing）

适用于：改 bug、改行为、改已上线接口字段等。

| 步 | 谁 | 做什么 |
|----|-----|--------|
| 1 | 你 | 在 `progress.md` 填 **变更暂停点**（`task_mode: change`），记下当前 `resume_ticket` |
| 2 | 你 | 从 `templates/` 复制模板到 `变更工单/CO-NNN_描述.md`（`hotfix` / `api-change` / `feature-change`） |
| 3 | 你 | 变更单里只写 **动机与期望**（阶段 0） |
| 4 | Agent 新会话 | 粘贴：`执行变更单 CO-NNN，阶段1只读，定位 API_CONTRACT/代码影响面` |
| 5 | 你 | 回复 `阶段1通过` → Agent 阶段 2 改代码 + 先改 docs（若 api-change） |
| 6 | 你 | 跑变更单里的验证命令 + `init.ps1` |
| 7 | Agent | 阶段 3；`progress.md` 恢复 `task_mode: linear` 和原主线 |

涉及 **路径/字段/SSE/错误码** → 必须用 **`api-change`**，且 **先改** `API_CONTRACT.md`、`ERROR_CODE.md`。

### 7.2 新增接口（新 REST 或新 SSE 事件）

| 步 | 做什么 |
|----|--------|
| 1 | 人类对齐：路径、方法、请求/响应、错误码、是否 SSE（可先讨论） |
| 2 | **先改 docs**：`API_CONTRACT.md`、`ERROR_CODE.md`，必要时 `DOMAIN_MAP.md` |
| 3 | 若仍在 F14–F17 工单范围内 → 在该工单阶段 2 实现；若超出 → 新建工单并更新 `work_orders.yaml` + `generate_orders.py --orders-only` |
| 4 | Agent 按工单/变更单阶段 1→2→3 执行 |
| 5 | 勾选 `docs/00-meta/API_CHANGE_CHECKLIST.md` |

**不要**在未更新 `API_CONTRACT.md` 的情况下让 Agent 「先写个 endpoint 试试」。

### 7.3 小 bug（不改对外契约）

走 **`hotfix` 变更单**，或当前工单阶段 2 末尾修（若该工单尚未标 passing）。仍要 pytest + `init.ps1`。

## 8. 什么时候开新会话（重要）

### 8.1 必须 / 强烈建议开新会话

| 时机 | 原因 |
|------|------|
| **上一张工单阶段 3 已完成**（已更新 handoff） | 下一张工单用新开场白，避免上下文串台 |
| **从「工单主线」切换到「变更单」** | 变更暂停点，避免 Agent 继续改 FNN |
| **从「变更单」回到主线** | `task_mode` 已改回 `linear` |
| **换了一个 Agent 工具** | 新工具读不到旧聊天，只能靠 handoff |
| **阶段 1 已通过，隔天才阶段 2** | 避免长对话漂移；新开并粘贴「阶段1通过，进入阶段2」 |
| **单工单阶段 2 对话已超过约 15–20 轮仍做不完** | 在 `session-handoff.md` 写断点，新开会话续做 |
| **`init.ps1` 或 pytest 在你这边失败，让 Agent 修** | 新开并附上失败日志，避免在旧代码假设上纠缠 |

### 8.2 可以不开新会话（同一对话继续）

| 时机 | 你发什么 |
|------|----------|
| 阶段 1 刚结束，进入阶段 2 | `阶段1通过，进入阶段2，严格按工单 Fxx §2.2` |
| 阶段 2 写完，等你本地验证 | 你先跑 pytest，再发 `验证通过，执行阶段3` |
| 阶段 3 中小修补丁 | 同一会话内修到 pytest 全绿 |

### 8.3 换工具但不换任务

1. 确认上一工具已写入 `session-handoff.md` + `progress.md`（否则先让它补）。
2. **新工具开新会话**。
3. 粘贴 handoff 开场白；若阶段 2 做了一半，加一句：`续工单 Fxx 阶段2，断点见 session-handoff.md`。

### 8.4 一张工单的推荐会话切分（省 token、更稳）

```text
会话A：阶段1 only → 你确认 → 结束
会话B：阶段2 实现 → 你本地验证 → 结束
会话C：阶段3 收尾 + 更新 handoff → 结束
```

比「一个会话干到底」更稳，尤其适合 F14、F15、F21 大工单。

## 9. 给 Agent 的通用话术模板（复制即用）

**开工（主线工单）** — 优先用 `session-handoff.md` 现成段落。

**阶段 2 放行**：

```text
阶段1通过，进入阶段2。严格按工单 {Fxx} §2.2 实现清单，遵守 ARCHITECTURE 分层，禁止整文件复制 TestNewHarness。
```

**阶段 3 收尾**：

```text
阶段2验证已通过：{粘贴 pytest/init 结果摘要}。执行阶段3，更新 feature_list、progress、session-handoff，下一任务指向 {Fyy} 阶段1。
```

**变更单**：

```text
按 CHANGE_WORKFLOW 执行变更单 CO-NNN，阶段1只读，输出影响文件与 API_CONTRACT diff，禁止写代码。
```

**附约束（任意工具建议加一句）**：

```text
遵守仓库 AGENTS.md；事实以 docs 与 feature_list.json 为准，不要凭记忆。
```

## 12. API 与前端对接（概要）

| 场景 | 接口 | 流式 |
|------|------|------|
| 编排对话（意图+工作流+Agent） | `POST /api/v1/run` | SSE |
| 纯聊天 | `POST /api/v1/chat` | SSE / 同步 |
| RAG 查询 | `POST /api/v1/kb/query` | SSE / 同步 |
| 知识库管理 | `/api/v1/kb/collections*` | 上传异步 task_id |
| 健康检查 | `GET /api/v1/health` | — |

选型见 `API_CONTRACT.md` **「API 选型决策表」**。  
Web 停止生成：**abort 当前 fetch**，不需 WebSocket（F06 实现断连检测后服务端停模型）。

错误码：REST 用整数 `code`；SSE `error` 事件用 `AI_2003` 格式（见 `ERROR_CODE.md`）。

## 15. 常见问题

**Q：`init.ps1` 失败？**  
先修环境（Python 包、pytest），不要在失败起点叠新功能。

**Q：F02 数据库测试连不上？**  
确认 `docker compose up -d postgres redis` 且 `DATABASE_URL` / `REDIS_URL` 与 compose 端口一致。

**Q：Agent 要不要复制老项目代码？**  
阶段 1 可读 `TestNewHarness` 对照；阶段 2 **按工单重写**，禁止整文件拷贝（表 3）。

**Q：下一会话还要重新解释 Harness 吗？**  
不用。只复制 `session-handoff.md` 开场白即可。

**Q：TestNewHarness 要上传吗？**  
默认 `.gitignore` 整目录；仅本地参考即可。
